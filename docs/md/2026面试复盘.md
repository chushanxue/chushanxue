## 高频简历深挖

### 系统与架构意识（核心主线）

#### 1、SSE和WebSocket的区别，选型依据

| 特性       | SSE (Server-Sent Events) | WebSocket                    |
| ---------- | ------------------------ | ---------------------------- |
| ⭐通信方向 | 单向（服务器→客户端）    | 双向 (全双工)                |
| ⭐协议     | 标准HTTP协议             | 独立WebSocket协议（基于TCP） |
| ⭐数据格式 | 仅支持UTF-8文本          | 支持文本和二进制数据         |
| ⭐自动重连 | 原生支持                 | 需要手动实现                 |
| 实现复杂度 | 简单                     | 较复杂                       |

在 POC 阶段，我们参考了 GitHub 上成熟的 AI 对话项目，使用 WebSocket 快速验证了多轮对话与实时推送能力

进入正式产品阶段后，发现 SSE 在单向流式推送上更符合当前业务，另外相比 WebSocket，SSE 可以直接复用现有的 HTTP 鉴权与中间件体系，也减少了前端在重连、状态恢复上的实现成本，因此最终选择将通信方案调整为 SSE。

#### 2、AI 输出不稳定前端怎么兜底

- 容错解析

  拿到结构化的流式数据后，主要是通过正则拆分以及判断是否为可解析的json格式，再进一步转为对象格式，通过不同的type匹配不同的UI展示，这一步也是 agent实现 的核心

  - `message_start / message`状态机决定了消息的生命周期
  - type 是 UI 展示的核心，不同 type 展示方式不同，这也是 agent 的核心
  - answerId 的存在，是“可恢复设计”的伏笔

- 断流问题

  把断流问题分成了三类：正常结束、网络异常、解析异常，这三类情况，本质上都表现为**前端流式解析无法继续推进**

  - 解决“断流带来的用户感知问题”（这块是自己写的，还需gpt验证）

    在断连场景下，前端不会直接销毁当前会话，而是把会话状态和已接收的流式内容通过localStorage缓存在浏览器当中。（那h5端怎么办），重连成功后，前端会基于已有状态继续渲染，而不是重新发起一轮对话，从用户视角看只是短暂的 loading，而不是功能中断。

    重连失败也会有兜底的UI提示和及时的状态变更

  - 解决“断流本身”

    首先SSE的主要特点包括了自动重连,如果连接断开，浏览器会自动尝试重新连接服务器。

    **（不需要主动提起）**当然自动重连机制虽然方便，但在某些情况下可能需要自定义重连策略或处理重连失败的情况。我们当时和业务方评估过实现成本和业务收益，业务侧并不存在“从中断点继续生成同一条回答”的强需求，因此前端并未引入自动重连机制，而是将异常结束视为可接受状态。如果后续业务对连续性提出更高要求，我们会考虑基于 answerId 引入重连策略和续写提示。

- 渲染问题

  大多数 Markdown 渲染库都是针对完整的 Markdown 文本输入设计的，它们没有“流式/中间态”安全机制，只要结构不完整（比如代码块 ``` 尚未闭合），就可能会导致渲染错误或意外结构

  实践上我们尝试过多种方案

  - 1、延迟渲染：在未闭合之前只做文本追加，不交给插件解析；当检测到结构闭合时，再把完整片段丢给插件渲染。
  - 2、通过视觉提示（如加载中图标、占位符）告知用户内容还在生成中。（延迟渲染）
  - 3、在 markdown 解析前，通过正则或语法分析检测不完整结构，添加临时闭合标签。

- 单段数据很大的话，可优化的方向？

  在流式对话场景下，前端性能瓶颈主要来自高频状态更新，所以前端会通过缓存 chunk、合并更新、降低解析频率来控制 render 次数，而不是每个数据片段都触发 UI 更新

  具体实现是用 useRef 缓存 chunk，到一定时间 / 字符数再 setState

- 顺序错乱（待验证以及websocket有没有这个问题，面试没问到就不用主动提及）

  - SSE 本身是单连接、单方向、有序推送
  - 前端是顺序 append，而不是并发渲染
  - 技术选型 + 实现方式共同决定不会遇到顺序错乱问题

### 工程与性能能力（可信度验证）

#### 1、Canvas 标注性能怎么扛住量级（大量图片、大量标注框）

用业务设计规避“假性能问题”，在系统设计阶段就约束了复杂度上限。（在当前业务模型下，Canvas 本身并不是瓶颈，主要性能风险来自交互频率与请求节奏，因此优先从系统层面进行控制，而不是过早引入复杂的渲染优化。）

实际使用场景中，真正的卡顿来源，是高频切换触发的接口请求堆积。在高频图片切换场景下，请求堆积导致主线程与网络资源竞争（这句不懂，有坑），引发明显的交互卡顿。

最开始的方案是限制图片切换频率（硬限制），但很快发现技术上可行，体验上不可接受，所以升级为基于用户停留行为的防抖请求策略。

这个项目里解决的不是“Canvas 如何扛住大量标注”，而是通过**产品思维**，实现“如何让系统永远不需要扛住大量标注”。

把标注框按标签分类，每次只展示一个标签下的标注框，由用户决定是否切换或多选展示其他标签下的标注框

标注框按标签维度可控渲染，同一时间绘制的标注框数量被严格控制，Canvas 重绘成本稳定

（如果用户强行勾选所有标签，标注很多怎么办？有过方案调研，会考虑引入分批渲染策略（这块可以再看看））

#### 2、你做过哪些性能优化

在项目中做性能优化时，我通常不会直接从代码层面入手，而是先判断性能问题的真实来源

- 第一步：问题归因，判断“是不是性能问题”，比如Canvas 项目里发生卡顿时，不是先优化 Canvas，而是发现卡顿来自高频请求堆积

- 第二步：判断“问题该在哪一层解决”，比如不为断流硬做重连、不为极端标注量硬做复杂渲染

- 第三步：优先用系统设计，而不是代码技巧，解决性能问题。常用手段包括：

  - 业务模型约束（单Canvas）
  - 数据规模约束（分页）
  - 可见性约束（标签切换）
  - 交互节奏控制（防抖）

#### 3、首屏渲染慢怎么优化

- 先判断首屏慢在「哪一段」

  - 网络（资源体积、请求数）
  - 解析执行（JS 体积、依赖复杂度）
  - 渲染（首屏组件复杂度）

- 根据项目形态选策略，而不是通用解法

  - ToB平台更关注交互可用性，不追求极致 TTI
  - ToC产品更关注首屏感知速度

- 落到具体手段
  - 首屏资源优先级、骨架屏
  - 异步组件、懒加载
  - 使用vite或webpack拆包策略

#### 4、vite和webpack

（准备两份配置文件并吃透）

Webpack 更偏向于生产构建阶段的工程化工具，而 Vite 更多是围绕开发体验和模块加载方式做的优化。

在实际项目中，Vite 在开发阶段的启动速度和热更新体验明显更好，尤其是在依赖较多的项目中，基于 ESM 的按需加载能显著降低启动成本；

Webpack 在构建阶段的配置灵活性和生态成熟度更高，适合对构建结果和兼容性要求更复杂的场景。
